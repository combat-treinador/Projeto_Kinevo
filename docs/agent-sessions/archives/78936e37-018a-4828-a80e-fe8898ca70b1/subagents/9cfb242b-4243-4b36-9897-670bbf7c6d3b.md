## 5. Offline / cache patterns

| Layer | Mechanism | Scope |
|-------|-----------|-------|
| **Firestore** | `memoryLocalCache()` + long polling in `firebaseService.ts` | Documents fetched while app is open remain readable offline; **no persistent disk cache** (RN lacks IndexedDB) |
| **Firebase Auth** | `getReactNativePersistence(AsyncStorage)` | Session survives app restarts |
| **React Query** | Default 5 min stale; per-query overrides (e.g. `useTodayPlan` `staleTime: 0`, history 30s) | In-memory query cache; cleared on student switch via `clearStudentSessionCache()` |
| **Zustand workoutStore** | In-memory only (no `persist` middleware) | Session progress lost on app kill |
| **SecureStore** | JWT access token | Auth for `apiClient` |
| **Plan source cache** | `_planSource` on plan response object in `workout.service.ts` | In-memory re-filter by prescription week |
| **Sound cache** | Module-level in `useCircuitBeep.ts` | AV asset reload |

No Firestore offline queue / optimistic write queue for workout saves — network required to persist history.

## 6. Technical debt & gaps

### Architecture / routing
- **Split `(main)/` vs `main/`** — tab routes are thin re-exports; real UI in `main/` creates confusion
- **`main/dashboard.tsx`** — mock biometrics chart, **not wired to any route**
- **`src/_deferred/`** — placeholder screens with TODOs; superseded by active `(auth)/` and `(main)/` but still in repo
- **`useTodayPlan`** marked legacy; Home uses **`useAllActivePlans`** instead

### Data & backend
- **`BibliotecaTreinos` read rules gap** — documented TODO in `workout.service.ts` (lines 23–36): client may not read library docs; plans return empty without Cloud Function or rule change
- **`apiClient`** fully built (`f:\Projeto_Kinevo\Kinevo_Alunos\src\services\apiClient.ts`) but **not used by active features** — all data via direct Firestore
- **Hardcoded Firebase config** in `firebaseService.ts` (TODO to move to `.env`)
- **Legacy field aliases** throughout history layer (`pseMedia`, `volumeTotal`, `exerciciosConcluidos`) — dual read paths in adapters

### Domain model
- **`planoMock.ts`** still defines canonical types (`Exercicio`, `PlanoDeTreino`, `DIAS_SESSAO`) used across production code despite name
- **`progressMock.ts`** — only `calcTrendPercent` used; rest is dead mock data

### UX / persistence
- **Workout progress not persisted locally** — kill app mid-session loses Zustand state
- **No true offline workout completion** — `saveCompletedWorkout` requires network
- **NativeWind installed but unused** in shipped screens — styling is 100% `StyleSheet`
- **Verbose `console.log` diagnostics** in `workout.service.ts` (production noise)

### Security / ops
- Firebase API keys committed in source
- Account link repair **deletes and recreates** `contasAlunos` doc when link missing (`studentLink.service.ts`) — fragile if rules/partial data edge cases

### Minor inconsistencies
- UI shows **RPE** label in one plans list string but collects **PSE** everywhere else
- **`date-fns` absent** — multiple ad-hoc date parsers across services (maintenance risk)
- Trainer role path works but adds complexity (impersonation, dual query paths on almost every service)
