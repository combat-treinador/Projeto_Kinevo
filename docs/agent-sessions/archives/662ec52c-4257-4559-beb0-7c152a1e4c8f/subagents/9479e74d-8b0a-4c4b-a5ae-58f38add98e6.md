## 4. React Query cache patterns

**Setup:** `QueryClient` in root `_layout.tsx` — default `staleTime: 5 min`, `retry: 2`.

**Query keys in use:**

| Key | Scoped by | Used in |
|-----|-----------|---------|
| `["todayPlan", userToken]` | token | `useTodayPlan`, invalidations |
| `["allActivePlans", userToken]` | token | `useAllActivePlans` |
| `["workoutHistory", userToken]` | token | hooks; **bug:** profile sync uses `auth.currentUser?.uid` instead |
| `["workoutHistoryByDate", dateISO]` | date only | `plans.tsx` |
| `["weeklyTimeSummary", userToken, weekStart]` | token + week | `(main)/index.tsx` |
| `["distanceCardioHistory", userToken]` | token (home) / none (progress) | inconsistent |
| `["personalRecords"]` | **not scoped** | `progress.tsx` |
| `["bodyWeightHistory"]` | **not scoped** | `progress.tsx` |
| `["studentDossier", "v2"]` | **not scoped** | `personal-data.tsx` |
| `["student-dossier-height"]` | **not scoped** | `progress.tsx` |

**Invalidation patterns:**
- **Prefix invalidation** — `invalidateQueries({ queryKey: ["todayPlan"] })` matches all `todayPlan*` keys
- **After workout completion** (`workout-session.tsx`, `plans.tsx`): invalidates `workoutHistory`, `workoutHistoryByDate`, `weeklyTimeSummary`, `todayPlan`, `allActivePlans`
- **Profile sync** (`(main)/profile.tsx`): invalidates + refetches `todayPlan` and `workoutHistory`
- **Logout:** only `removeQueries(["todayPlan"])` — other caches may linger until remount

**Gap for impersonation:** several progress/dossier queries are not keyed by student scope; switching students would show stale/wrong data without a full cache reset.

## 5. Trainer impersonation — current state and integration points

**Nothing exists yet:** no matches for `activeStudentId`, `StudentSelectorScreen`, or `impersonat` anywhere under `f:\Projeto_Kinevo`.

The app is architected as **student-only**: identity always flows `auth.currentUser.uid → contasAlunos → { studentId, ownerUid }`. A trainer logging in with their web credentials would have a different UID and likely no `contasAlunos` student document.

### Recommended integration points

**A. Extend `AuthContext` (`AuthContext.tsx`)**

Add to `AuthState`:
```typescript
userRole: 'student' | 'trainer' | null;
activeStudentId: string | null;   // kp_xxx when impersonating
activeStudentName: string | null;
isImpersonating: boolean;
```

Methods: `setActiveStudent(studentId)`, `clearActiveStudent()`, `listTrainerStudents()`.

Detect role via Firebase custom claims (`getIdTokenResult()`) or a `perfisUsuario/{uid}` doc (web trainer profile).

**B. Centralize student resolution**

Refactor all services to use one function, e.g. extend `resolveStudentLink()` in `assessment.service.ts`:

```typescript
// Pseudocode
if (impersonationContext.activeStudentId) {
  return { studentId: activeStudentId, ownerUid: auth.currentUser.uid, ... };
}
// else existing contasAlunos logic
```

**Must also update** `workout.service.ts` (`fetchTodayPlan`, `fetchAllActivePlans`, `saveCompletedWorkout`, `fetchWorkoutHistory`) — it bypasses `resolveStudentLink()` today.

**C. `StudentSelectorScreen` + routing guard**

New route: `f:\Projeto_Kinevo\Kinevo_Alunos\src\app\(trainer)\student-selector.tsx` (or `(main)/student-selector.tsx`).

Update `NavigationGuard` in `_layout.tsx`:

```
if (!userToken) → (auth)/login
else if (userRole === 'trainer' && !activeStudentId) → student-selector
else → (main)
```

Add a persistent banner in `(main)/_layout.tsx` when `isImpersonating` ("Viewing as {name} · Switch student").

**D. React Query scoping**

Include effective student in all keys:
```typescript
queryKey: ["todayPlan", userToken, activeStudentId ?? "self"]
```

On student switch:
```typescript
queryClient.clear(); // or removeQueries for all known prefixes
useWorkoutStore.getState().resetWorkoutProgress();
```

Fix inconsistent keys in `progress.tsx` and `personal-data.tsx`.

**E. Firestore security rules (blocker)**

`workout.service.ts` documents that `BibliotecaTreinos` read rules may not allow mobile clients. Trainer impersonation additionally requires rules allowing a trainer UID to read/write student-scoped collections (`workouts`, `historicoTreinos`, `recordesPessoais`, etc.) when `ownerUid == request.auth.uid`. Without rule changes, impersonation will fail at runtime regardless of client code.

**F. Write operations**

`saveCompletedWorkout` sets `studentUid: user.uid` (trainer UID when impersonating). Decide whether history writes should use the student's Firebase UID or a trainer-audit field — affects web dashboard display.
