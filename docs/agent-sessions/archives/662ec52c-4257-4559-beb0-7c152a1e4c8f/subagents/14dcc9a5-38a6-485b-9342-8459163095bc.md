## Architecture Summary

There is **no `planStore`**. Plan data lives in **React Query cache** + **AsyncStorage plan cache**. Session/progress state lives in **`workoutStore`** (Zustand persist).

```
Firestore (workouts + BibliotecaTreinos)
    ↓ subscribeActiveWorkoutPlans / fetchAllActivePlans / fetchTodayPlan
workout.service.ts (adaptBibliotecaToResponse)
    ↓ persistActivePlansResult
planCache.service.ts (AsyncStorage, schema v5)
    ↓ queryClient.setQueryData
useActivePlansLiveQuery → useAllActivePlans / useTodayPlan
    ↓ syncPlanFromServer (IDs only)
workoutStore.ts (activePlanId, completedSessionIdsByPlan — NOT plan exercises)
```

## Recommended Fix Locations

Depending on the bug:

| Symptom | Fix location |
|---------|--------------|
| Exercises missing after Web edit | `dedupeRawExercicios` / `normalizarSemanasExercicios` / `filterExerciciosForWeek` in `workout.service.ts` + `exercisePrescription.ts` |
| Duplicate exercises in UI | `dedupeExercicios` (mapped) or `dedupeRawExercicios` (raw) |
| Stale plan after realtime update | `subscribeActiveWorkoutPlans` fingerprint/debounce (`fingerprintActivePlansResult`, `activePlansResultsEqual`) + `useActivePlansLiveQuery.cloneActivePlansResult` |
| Offline cache not refreshing | `persistActivePlansResult` — verify `clearAllPlanCaches` + fingerprint change |
| Wrong week exercises | `rebuildPlanForPrescriptionWeek` + `_planSource.exercicios` |
| Plan ID out of sync | `syncPlanFromServer` / `clearActivePlanState` in `workoutStore.ts` + effects in `useAllActivePlans.ts:86–103` |
| Inactive/deleted plans still showing | `selectValidWorkoutAssignmentsByBiblioteca`, `isWorkoutAssignmentClearlyInvalid`, filter in `useAllActivePlans.ts:157–167` |
| Session exercises stale mid-workout | `bindActiveSessionCatalog` / `activeSessionCatalog` in `workoutStore.ts` (frozen at workout start) |
| Bump cache after adapter contract change | Increment `PLAN_CACHE_SCHEMA_VERSION` in `planCache.service.ts` |

**Most likely root cause for "ghost/stale exercises":** the pipeline is **replace-only** at every layer; there is no merge-with-previous-plan logic. Stale data usually comes from (a) dedup dropping items, (b) fingerprint skipping emit, (c) frozen `activeSessionCatalog` during an active session, or (d) old AsyncStorage envelope with mismatched schema version being silently discarded while React Query still holds prior in-memory data.
